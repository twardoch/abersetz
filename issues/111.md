
## Task 111

Prepare a comprehensive overhaul of the engine selector syntax. Write the plan into ./TODO.md 

I want to be able to specify the engine and provider like: 

1. engine (2-letter, followed by `::`): 

- `tr::` translators 
- `dt::` deep-translator
- `lm::` LMStudio Python SDK
- `ll::` OpenAI-compatible LLM API
- `ml::` local MLX model with direct Python MLX_LM
- `gg::` local GGUF model with direct Python Llama.cpp

It’s possible to specify a subvariant of the engine after the slash: `lm/gemma::` or `ml/hy-mt2::`

2. engine::provider 
- `tr::apertium`
- `tr::argos`
- `tr::...`
- `dt::google`
- `lm::{LMStudio model identifier}`
- `ll::{provider_endpoint_identifier}:{model_identifier}`
- `ml::{/path/to/model/folder}`
- `gg::{/path/to/model/file.gguf}`

3. Support a job JSON format, which should specify engine-provider-models and their parameters, and an output naming scheme (slugifed suffixes that identify the engine-provider-model-config). It can take a list of multiple configs. 

4. Combine 'engines' and 'discover' into one consistent 'ls' CLI command that lists all available engines, providers & models, or, if only the beginning of the selector is specified (optionally ), lists all available engines, providers & models that match the beginning of the selector, or if wildcard is specified, only those that match the wildcard. If the subset is specified, it should be a fast operation. For example normally we’d query APIs for their model lists, and we don’t want to do it if APIs are not asked. Also for SLOW operations we should cache the results, and use cache unless --force is used.  

If --job is given, the output should be an abersetz job JSON format.

Each of these should take --job JSON as input for configuring translation(s).
